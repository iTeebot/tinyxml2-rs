# ADR-0003: Parser Design

| Field   | Value                |
|---------|----------------------|
| Status  | **Accepted**         |
| Date    | 2026-06-01           |
| Authors | tinyxml2-rs maintainers |

---

## Context

The parser is the most critical component of tinyxml2-rs. It must:

1. **Exactly match TinyXML2's parsing behavior** for all inputs — including malformed
   XML, edge cases, and error conditions
2. **Produce the same DOM structure** as TinyXML2 for any given input
3. **Report the same errors** in the same conditions
4. **Handle TinyXML2-specific quirks** that may deviate from the XML 1.0 specification

TinyXML2's parser is itself a hand-written recursive descent parser in C++. It has
numerous behavioral quirks that are well-established and depended upon by downstream
consumers. These quirks include specific whitespace handling modes, entity processing
options, error precedence rules, and tolerance for certain malformed inputs.

## Problem Statement

How do we implement an XML parser that achieves exact behavioral compatibility with
TinyXML2, including all edge cases and error conditions?

## Alternatives Considered

### Alternative 1: Parser Generator (pest, lalrpop)

Define a formal grammar and use a parser generator to produce the parser.

```pest
// Example pest grammar
document = { SOI ~ prolog? ~ element ~ EOI }
element  = { "<" ~ name ~ attribute* ~ ("/>" | ">" ~ content* ~ "</" ~ name ~ ">") }
```

**Pros:**
- Declarative grammar definition — easy to read and validate
- Automatically handles parsing mechanics (tokenization, backtracking)
- Grammar is self-documenting
- Mature tooling and ecosystem

**Cons:**
- **Limited error control**: Error messages are generated by the framework and are
  often cryptic or unhelpful for end users
- **Behavioral matching is difficult**: TinyXML2's quirks (e.g., tolerating certain
  malformed inputs, specific whitespace handling) are hard to express in a formal grammar
- **Build dependency**: Parser generators add compile-time dependencies and may slow
  builds
- **Abstraction leak**: When the grammar doesn't match desired behavior, workarounds
  are needed that fight the abstraction
- **Line tracking complexity**: Integrating line/column tracking into generated parsers
  requires extra effort

### Alternative 2: Parser Combinator (nom, combine, winnow)

Build the parser by composing small parsing functions.

```rust
// Example nom parser
fn element(input: &str) -> IResult<&str, Element> {
    let (input, _) = char('<')(input)?;
    let (input, name) = take_while1(is_name_char)(input)?;
    // ...
}
```

**Pros:**
- Composable and testable at the function level
- Good for binary/mixed formats
- No build-time code generation
- Reasonable error handling with custom error types

**Cons:**
- **Complexity accumulates**: XML's grammar (especially with entities, CDATA, comments,
  and declarations) leads to deeply nested combinators that are hard to follow
- **Error messages**: While better than parser generators, still requires significant
  effort to produce user-friendly error messages
- **Behavioral matching**: Matching TinyXML2's exact behavior requires escape hatches
  from the combinator pattern
- **Adds a dependency**: `nom` or similar becomes a foundational dependency
- **Debugging**: Tracing parser behavior through combinator chains is challenging
- **Performance**: Combinator overhead (closures, trait objects) may impact hot paths

### Alternative 3: Hand-Written Recursive Descent

Write the parser manually as a set of recursive functions that directly consume the
input string.

```rust
impl Parser {
    fn parse_element(&mut self) -> Result<NodeId, XmlError> {
        self.expect('<')?;
        let name = self.parse_name()?;
        let attributes = self.parse_attributes()?;
        // ...
    }
}
```

**Pros:**
- **Full control**: Every parsing decision is explicit and auditable
- **Line-by-line behavioral matching**: Can replicate TinyXML2's C++ parser logic
  directly, including quirks and edge cases
- **Custom error messages**: Error messages can reference exact positions and provide
  context-specific guidance
- **Line/column tracking**: Trivially integrated into the scanning loop
- **No dependencies**: Zero external crates
- **Debuggable**: Standard Rust debugging tools work directly on the parser
- **Single-pass construction**: Can build the DOM directly in the arena during parsing
  — no intermediate AST required

**Cons:**
- **More code**: Must implement all parsing logic manually (~500-1000 lines)
- **No formal grammar**: Correctness is validated through testing, not grammar analysis
- **Maintenance burden**: All XML constructs must be handled explicitly
- **Risk of bugs**: No framework safety net — parser bugs are our responsibility

## Decision

**Option 3 — Hand-written recursive descent parser with single-pass DOM construction.**

## Reasoning

1. **Behavioral fidelity is paramount.** The project's #1 priority is correctness, and
   #2 is compatibility. A hand-written parser provides the control needed to match
   TinyXML2's behavior in every edge case, including:
   - How it handles malformed XML
   - When it stops parsing on error
   - How whitespace is normalized in different modes
   - How entities are processed (or not) based on configuration
   - How self-closing tags are handled
   - Error precedence rules

2. **TinyXML2's own parser is recursive descent.** Matching its behavior is most natural
   when using the same parsing strategy. We can compare our parser's logic against the
   C++ source line-by-line.

3. **Single-pass construction.** The parser builds the DOM directly in the generational
   arena as it parses. There is no intermediate AST — this reduces memory allocation,
   eliminates a translation step, and simplifies the architecture.

4. **No dependencies.** Consistent with the project's zero-dependency goal.

5. **Custom error messages.** Error messages include line numbers and contextual
   information, matching or improving upon TinyXML2's error reporting.

## Consequences

### Parser Architecture

```
Input (&str)
    │
    ▼
┌─────────────────────────────────────────┐
│  Parser                                 │
│  ├── cursor: &str  (remaining input)    │
│  ├── line: u32     (current line)       │
│  ├── arena: &mut Arena<Node>            │
│  └── options: ParseOptions              │
│      ├── process_entities: bool         │
│      ├── whitespace_mode: WhitespaceMode│
│      └── max_depth: u32                 │
└─────────────────────────────────────────┘
    │
    ▼
Arena<Node>  (DOM built in-place)
```

### Parsing Flow

```
parse_document()
├── parse_bom()               // Consume optional UTF-8 BOM
├── parse_declaration()       // <?xml ...?>
└── parse_node() ◄──────┐    // Dispatch based on next chars
    ├── parse_element()  │    // <name ...> ... </name>
    │   ├── parse_attributes()
    │   └── parse_node() ┘    // Recursive: children
    ├── parse_comment()       // <!-- ... -->
    ├── parse_cdata()         // <![CDATA[ ... ]]>
    ├── parse_text()          // Character data
    └── parse_unknown()       // <! ... >
```

### Key Design Decisions Within the Parser

| Decision | Rationale |
|----------|-----------|
| No intermediate AST | Single-pass construction reduces allocations and simplifies architecture |
| Cursor is `&str` slice | Zero-copy scanning; substring operations are O(1) |
| Line tracking in scan loop | Accurate error positions with minimal overhead |
| Error stops parsing | Matches TinyXML2 behavior: first error is fatal, no recovery |
| Depth counter | Prevents stack overflow from deeply nested or malicious XML |
| Entity processing is configurable | Matches TinyXML2's `processEntities` flag |
| Whitespace mode is configurable | Matches TinyXML2's `PRESERVE`/`COLLAPSE`/`PEDANTIC` modes |

### Error Handling

The parser maps errors to the `XmlError` enum, which corresponds to all TinyXML2
`XMLError` codes:

```rust
pub enum XmlError {
    NoAttribute,
    WrongAttributeType,
    FileNotFound,
    FileCouldNotBeOpened,
    FileReadError,
    ElementMismatch,
    ParsingElement,
    ParsingAttribute,
    ParsingText,
    ParsingCData,
    ParsingComment,
    ParsingDeclaration,
    ParsingUnknown,
    EmptyDocument,
    MismatchedElement,
    Parsing,
    CountMismatch,
    NoTextNode,
    NoExistingAttribute,
    ElementDepthExceeded,
    // ...
}
```

### Trade-offs Accepted

| Trade-off | Justification |
|-----------|---------------|
| More code to write | ~500-1000 lines, but each line is purposeful and auditable |
| No formal grammar | Correctness validated by 87+ tests (Phase 1) growing to hundreds |
| Tight coupling to arena | Intentional: single-pass construction is a feature, not a bug |
| Manual maintenance | All changes are explicit; no hidden framework behavior |

### Testing Strategy

1. **Unit tests**: Test each parsing function in isolation
2. **Integration tests**: Parse complete documents and verify DOM structure
3. **Conformance tests**: Port TinyXML2's own test suite
4. **Round-trip tests**: Parse → serialize → parse → compare
5. **Fuzz tests**: Random/mutated inputs to find parser crashes or panics
6. **Edge case tests**: Catalog of known edge cases with expected behavior
